# Architecture

Heliosnet is a mission-operations system for a fictional satellite constellation.
The repos are independent and each stands alone, but they share one telemetry
schema, one anomaly taxonomy (`nominal`, `drift`, `dropout`, `thermal_runaway`),
and one operating principle: no command reaches a vehicle without a human
approving it.

## Shared contracts

- **Telemetry schema.** 14 channels across power, thermal, ADCS, link, and
  payload subsystems. `constellation` defines it; `constellation-stream`,
  `groundstation`, and `groundstation-train` vendor a compact copy so each repo
  reproduces from a clean clone with no cross-repo import.
- **Anomaly taxonomy + gate policy.** Severity thresholds, the remediation map,
  and the approval/emission rules are the same everywhere. `groundstation`
  enforces them at run time; `groundstation-train` vendors them as the single
  source of ground truth for both its training data and its eval.
- **Approval gate.** Any command other than a no-op monitor requires explicit
  operator approval and is never reported as emitted without it. This is the
  behaviour the console surfaces, the copilot enforces, the trained core is
  aligned to, and `aegis` authenticates.

## Honesty model

Each repo carries the same three-way split:

- **Real**: runs from a clean clone, exercised in CI, produces the committed
  numbers.
- **Simulated**: the inputs (telemetry, imagery, corpus) are deterministically
  generated so the repos are self-contained and offline, with ground-truth labels.
- **Documented, not run here**: anything needing hardware the build cannot
  provide (a GPU fine-tune, a billion-row cluster run, a live cloud apply) is
  given as an exact command and clearly labeled, never claimed as an executed
  result.

The meta-repo metrics table is generated by `tools/collect_metrics.py`, which
reads each repo's committed artifact and reports only what exists on disk.
